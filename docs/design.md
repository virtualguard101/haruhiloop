# HaruhiLoop 设计文档

本文档基于当前代码实现（`v2.3.x`）整理项目的设计目标、架构分层、玩法机制与关键技术决策，作为后续迭代的统一参考。

## 1. 项目定位

`HaruhiLoop` 是一个受《凉宫春日的忧郁》“无尽的八月”启发的时间循环模拟器，提供：

- `haruhi`：面向脚本/命令链的 CLI 入口（Typer）
- `haruhi-play`：面向沉浸体验的键盘交互 TUI（Textual + Rich）

核心体验不是“动作操作”，而是“在重复日常中通过微小选择改写系统状态与叙事走向”。

## 2. 设计理念

### 2.1 状态优先（State-first）

- 游戏推进本质是状态机演化，不依赖临时 UI 状态。
- 单步输入（scene + choice）必须映射到可落盘、可回放的状态变化。
- 当前在确定性模式下，同样初始条件 + 同样动作序列，结果可复现。

### 2.2 系统-叙事双轨

- 系统层：满意度、稳定度、线索、疲劳等硬指标提供可计算反馈。
- 叙事层：动作风味文本、结局剧情、视觉氛围面板提供语义解释。
- 两层共同服务“循环压迫感”和“世界线修复感”。

### 2.3 可演进的工程化分层

- 对外入口保持稳定（兼容旧路径）。
- 逐步将编排逻辑、领域逻辑、持久化、渲染从“单模块实现”迁移到分层包。
- 允许在过渡期“新层 + 旧门面”并存，降低重构风险。

### 2.4 可观测与可回放

- 每一步均写入 `history.jsonl`，支持回看“为什么走到这个结局”。
- 状态快照写入 `state.json`，支持恢复、继续、批量模拟对比。

## 3. 架构设计（当前）

当前采用“分层结构 + 兼容门面”：

- `interfaces`：CLI/TUI 入口与渲染适配
- `application`：用例编排服务（`start / step / status / replay / simulate`）
- `domain`：规则、引擎、模型、子系统
- `infrastructure`：持久化（文件存档）
- `narrative`：i18n、动作文案、结局文本

同时保留历史模块路径（如 `play_app.py`、`view.py` 等）作为兼容层，保证脚本与测试平滑过渡。

> 详细分层参考：`docs/arch.md`

## 4. 玩法模型

### 4.1 时间推进

- 时间槽：`morning -> afternoon -> evening`
- 傍晚结束后进入下一天，并在条件满足时体现循环感与世界线变化。

### 4.2 核心状态

每个 run 至少包含以下关键状态：

- 节奏与进度：`day`、`timeslot_index`、`loop_count`
- 主指标：`satisfaction`、`stability`、`clue_points`
- 危机与子系统：`closed_space_stage`、`homework_progress`、`crew_sync`
- 暗线与叙事：`nagato_fatigue`、`flags`、`ending_id`、`ending_epilogue`
- 扰动机制：`mutator_mode`、`worldline_mutation_profile`

### 4.3 单步交互循环

1. 选择场景（scene）
2. 选择选项（choice）
3. 引擎计算增量、触发事件与结局判定
4. 写入历史记录
5. 更新状态快照
6. 进入下一步选择或结局页

### 4.4 结局设计

- 结局不是单一阈值触发，而是多状态耦合后的叙事分流。
- 通过“高压崩坏 / 协调收束 / 代价性突破”等方向形成差异化收束体验。

## 5. 存档与持久化设计

### 5.1 存储结构

默认存放在工作目录：

```text
.haruhiloop_runs/<run_id>/
  state.json
  history.jsonl
```

### 5.2 设计原则

- `state.json`：当前可恢复状态（快速继续）
- `history.jsonl`：逐步历史（可追踪与复盘）
- schema 版本校验：避免不兼容存档静默损坏

### 5.3 启动界面加载策略（当前实现）

- 开始菜单仅保留入口选项（新局、载入、帮助、退出）
- 存档加载拆分为独立页面，避免大量存档挤占主菜单
- 存档按最后修改时间排序，编号形如 `YYYYMMDD-序号`
- 存档页支持分页与快捷键加载（适配大量存档场景）

## 6. TUI 交互与视觉设计

### 6.1 交互层次

- 入口态（entry）
- 存档选择态（load）
- 游戏态（game）

这种显式屏态划分可以保持输入语义清晰，避免按键冲突。

### 6.2 视觉风格

- 主体采用 Rich 面板分区，强调“信息层次而非复杂图形”
- 引入 ASCII 角色图 + Logo 形成题面识别度
- 使用凉宫主题文案与 Galgame 风格菜单框提升氛围一致性

### 6.3 信息设计

- 游戏态优先展示：世界线状态、关键台词层、一步反馈、状态面板、选择器
- 入口态优先展示：品牌感与明确入口路径
- 存档态优先展示：可选项密度、页码、操作提示

## 7. CLI 与 TUI 的职责分工

- CLI：强调自动化、可脚本化、批量模拟与测试友好
- TUI：强调沉浸感、渐进反馈、键盘内闭环操作

两者共享同一状态模型与持久化机制，保证结果语义一致。

## 8. 非功能性目标

- 可测试：核心规则可用单元测试覆盖
- 可兼容：旧导入路径与入口命令可继续使用
- 可维护：领域逻辑与呈现逻辑逐步解耦
- 可扩展：后续可继续增加子系统、结局线与 UI 屏态

## 9. 后续演进建议

1. 将 `play_app.py` 的复杂流程逐步迁入 `interfaces/tui/*`，旧文件仅保留门面。
2. 为存档选择页加入“高亮游标 + Enter 确认”模式，统一 Galgame 操作体验。
3. 在 application 层增加显式的 SaveSlot 用例服务，减少 UI 直连 storage。
4. 以文档驱动方式维护“状态字段 -> 机制 -> 结局条件”的映射表，降低认知负担。
